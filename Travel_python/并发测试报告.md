# Agent 项目并发分析报告

## 一、并发场景梳理

### 场景 1：Web API 多请求并发

**入口**：`app/web/api.py` → `@router.post("/chat")`

**路径**：

```
请求A (/api/chat) ──┐
请求B (/api/chat) ──┼──→ process_with_agent() ──→ multi_agent.ainvoke()
请求N (/api/chat) ──┘
```

**描述**：FastAPI 支持同一进程并发处理多个 HTTP 请求，每个请求独立调用 `process_with_agent()`，进而调用 LangGraph 的 `multi_agent.ainvoke()`。

这个的主要问题是session_id，但是这个session_id是从Java那边生成的，所以不会出现不同用户使用同一个session的情况

---

### 场景 2：同一 Session_ID 的并发读写

**入口**：用户快速连续发送两条消息，使用相同的 `session_id`

**相关代码** (`app/utils/redis_client.py`)：

```python
def get_session_context(session_id: int) -> list:
    data = redis_client.get(key)      # ← 读
    return json.loads(data) if data else []

def save_session_context(session_id: int, messages: list):
    redis_client.set(key, json.dumps(messages), ...)  # ← 写
```

**并发路径**：
```
请求A: get_session_context(8888)  → []
请求B: get_session_context(8888)  → []
请求A: save_session_context(8888, [A的消息])
请求B: save_session_context(8888, [B的消息])  ← A的消息被覆盖！
```

**影响**：用户消息历史丢失或不完整。

---

### 场景 3：同一 User_ID 的 Profile 并发更新

**入口**：同一用户的多个请求触发后台 Profile 更新

**相关代码** (`app/service/multi_agent_travel.py`)：

```python
def _update_profile_background(user_id: int, task: TravelTask, ...):
    loop.run_in_executor(None, functools.partial(_update_profile_from_task, ...))

def _update_profile_from_task(user_id: int, task: TravelTask, ...):
    profile = get_user_profile(user_id)   # ← 读（无锁）
    # ... 基于读到的数据做修改 ...
    save_user_profile(user_id, profile)   # ← 写（无锁）
```

**并发路径**：多个后台线程同时读写 `user:profile:{user_id}`，后写入覆盖先写入。

**影响**：用户偏好（饮食禁忌、喜欢的食物、住宿偏好、旅行风格等）数据丢失。

---

### 场景 4：MQ 消息消费者 + Web API 并发

**入口**：
- `MessageConsumer`（RocketMQ 消费消息）→ `process_with_agent()`
- `api.py` 的 `/api/chat`（HTTP 请求）→ `process_with_agent()`

**描述**：两套入口可能同时处理同一个 `user_id` 或 `session_id` 的请求，在 Redis 读写和 Profile 更新时产生冲突。

---

### 场景 5：Mem0 长期记忆并发写入

**入口**：`app/utils/mem0_client.py` → `add_memory_background()`

```python
# 问答流程
add_memory_background([user_msg, assistant_answer], user_id)

# 规划流程
add_memory_background([user_msg, assistant_plan_msg], user_id)
```

**描述**：多个 `run_in_executor` 线程同时调用 `memory.add()`，Mem0 内部 Qdrant 向量数据库的写入可能产生竞争。

---

### 场景 6：LangGraph 多 Agent 节点内部串行

**入口**：`multi_agent.ainvoke()` 内部

```
researcher ──┐
planner    ──┼──→ supervisor
critic     ──┘       ↑
              refiner ──→ critic（循环）
```

**描述**：单个 `ainvoke()` 调用内部按图拓扑顺序串行执行，不存在节点间的并发问题。但多个并发的 `ainvoke()` 调用各自有独立状态副本，共享的 Redis 状态存在竞态。

---

### 场景 7：Redis 全局连接对象并发使用

**入口**：所有 Redis 操作

```python
redis_client = redis.from_url(settings.REDIS_URL, decode_responses=True)
```

**说明**：`redis-py` 的连接池是线程安全的，但业务层的 `get → modify → set` 模式不是原子操作，并发下会互相覆盖。

---

## 二、问题总结

### 问题 1：Redis 会话上下文并发覆盖（高风险）

**根因**：读-改-写三步操作不是原子的。

```
T1: read session_8888  → []
T2: read session_8888  → []
T1: write session_8888 → [msg_A]
T2: write session_8888 → [msg_B]  # msg_A 丢失
```

**触发条件**：用户快速连续发消息，或同一 session_id 被多个请求共享。

---

### 问题 2：用户 Profile 并发更新数据丢失（高风险）

**根因**：同问题1，`get → 修改 → set` 模式在多线程环境下不安全。

```python
# 线程A              # 线程B
get_profile()        get_profile()
→ {budget: "中等"}   → {budget: "中等"}
modify: 加 preferred_destinations
→ {..., "preferred_destinations": ["东京"]}
save_profile()       modify: 加 travel_style
                     → {..., "travel_style": ["亲子"]}
                     save_profile()  # 线程A的 preferred_destinations 丢失！
```

**触发条件**：同一 user_id 的多个请求几乎同时完成并触发后台更新。

---

### 问题 3：Mem0 并发写入冲突（中等风险）

**根因**：Mem0/Qdrant 多线程写入可能产生向量数据库层面的竞争。

**触发条件**：多个请求的对话几乎同时结束，同时触发 `add_memory_background()`。

---

## 三、解决方案

### 方案 1：Redis 操作原子化（核心修复）

#### 方案 A：使用 Redis 事务（MULTI/EXEC）

```python
# 替换 get + set 为原子操作
def append_session_message(session_id: int, role: str, content: str):
    """原子地追加一条消息到会话上下文"""
    key = f"chat:session:{session_id}"
    # 使用 Lua 脚本保证原子性
    script = """
    local data = redis.call('GET', KEYS[1])
    local messages = data and cjson.decode(data) or {}
    messages[#messages + 1] = {role = ARGV[1], content = ARGV[2]}
    redis.call('SET', KEYS[1], cjson.encode(messages), 'EX', ARGV[3])
    return #messages
    """
    redis_client.eval(script, 1, key, role, content, settings.REDIS_SESSION_TTL)
```

#### 方案 B：使用 Redis Hash 替代 String + JSON

```python
def append_session_message(session_id: int, role: str, content: str):
    """使用 Hash 存储，每条消息一个 field，避免整体覆写"""
    key = f"chat:session:{session_id}"
    import time
    msg_id = str(int(time.time() * 1000000))  # 微秒级时间戳作为 field
    pipe = redis_client.pipeline()
    pipe.hset(key, msg_id, json.dumps({"role": role, "content": content}))
    pipe.expire(key, settings.REDIS_SESSION_TTL)
    pipe.execute()
```

#### 方案 C：Profile 更新使用 Redis Hash 字段级操作

```python
def add_user_preference(user_id: int, pref_type: str, value: str):
    """原子地追加单一偏好类型，不影响其他字段"""
    key = f"user:profile:{user_id}"
    # 使用 sadd 管理列表类型偏好
    if pref_type in ("preferred_destinations", "preferred_budgets", "preferred_paces",
                     "dietary_restrictions", "preferred_foods", "travel_style"):
        redis_client.sadd(f"{key}:{pref_type}", value)
    else:
        redis_client.hset(key, pref_type, value)
```

### 方案 2：Profile 更新加分布式锁

如果方案1实现成本高，可以加一个简单的分布式锁：

```python
import uuid

def update_user_profile_locked(user_id: int, task: TravelTask, final_plan_text: str):
    """使用 Redis SET NX 实现分布式锁"""
    lock_key = f"lock:profile:{user_id}"
    lock_value = str(uuid.uuid4())

    # 尝试获取锁，最多等 5 秒，锁自动 30 秒过期
    acquired = redis_client.set(lock_key, lock_value, nx=True, ex=30)
    if not acquired:
        logger.warning(f"Profile 更新被跳过（锁未获取）: user_id={user_id}")
        return

    try:
        # 执行更新逻辑
        _update_profile_from_task(user_id, task, final_plan_text, [])
    finally:
        # 释放锁（只释放自己持有的锁）
        if redis_client.get(lock_key) == lock_value:
            redis_client.delete(lock_key)
```

### 方案 3：Mem0 并发写入控制

```python
import asyncio
from threading import Semaphore

# 限制最多 3 个并发写入
_mem0_semaphore = Semaphore(3)

def add_memory_background(messages: list, user_id: int):
    """限制并发数，避免 Qdrant 写入过载"""
    if messages and len(messages) > 0:
        loop = asyncio.get_event_loop()
        loop.run_in_executor(
            None,
            functools.partial(_do_add_memory_with_limit, messages, user_id)
        )

def _do_add_memory_with_limit(messages: list, user_id: int):
    with _mem0_semaphore:
        _do_add_memory(messages, user_id)
```

### 方案 4：Web API 层加请求合并（可选优化）

对于同一 session_id 的连续快速请求，可以在 API 层做简单合并：

```python
from fastapi import APIRouter
from collections import defaultdict
import asyncio

_pending_sessions = defaultdict(list)

@router.post("/api/chat")
async def chat(request: ChatRequest):
    session_id = request.session_id or int(time.time() * 1000) % 1000000

    # 如果该 session 已有请求在处理中，排队等待
    if _pending_sessions[session_id]:
        future = asyncio.Future()
        _pending_sessions[session_id].append(future)
        result = await future  # 等待前一个请求完成
        return result

    # 第一个请求，开始处理
    _pending_sessions[session_id].append(None)
    try:
        result = await _do_chat(request, session_id)
        return result
    finally:
        # 唤醒所有排队的请求
        pending = _pending_sessions.pop(session_id, [])
        for f in pending:
            if f:
                f.set_result(result)
```

## 五、总结

| 问题 | 根因 | 解决方案 |
|-----|------|---------|
| Session 消息覆盖 | get-set 非原子 | Redis Hash / Lua 脚本原子化 |
| Profile 偏好丢失 | 多线程 get-modify-set | Redis Hash 字段级操作 / 分布式锁 |
| Mem0 写入冲突 | 多线程并发调用 | Semaphore 限流 |
| MQ + Web 数据竞争 | 共享 Redis 状态无隔离 | 同上，原子操作 + 锁 |



解决方案：

1、和deepseek类似，用户点击发送按钮后，后台agent处理问题，但是前端页面发送按钮位置就会换成停止按钮，点击停止按钮后台就会停止处理问题。就不会出现一次性发送多条信息，导致agent处理错乱。

2、当一个用户打开了多个session时，完成行程规划的时候，需要更新profile时，就会出现问题，可能更新会被覆盖。（可以使用锁来处理）
