## 项目核心流程（状态机视角）

以下流程以用户一次完整的旅游攻略生成请求为例，详细描述从前端到后端、从 Java 到 Python、再到结果返回的全链路状态变化与交互细节。

### 1. 用户输入与前端处理
- **用户操作**：在微信小程序中通过**文字输入**或**语音录入**提交旅游想法（例如：“我想去北京玩3天，喜欢历史古迹和美食”）。
- **前端动作**：
  - 若为语音，先调用微信小程序语音识别 API 转成文本。
  - 生成前端请求 ID（可选），封装 JSON 请求体：`{ "userId": "xxx", "query": "我想去北京...", "inputType": "text" }`。
  - 调用 Java 后端的 REST API（例如 `POST /api/travel/plan`）。
  - 同时，前端建立 WebSocket 连接（STOMP 协议），订阅主题 `/user/{userId}/task/{taskId}`，用于接收实时进度和最终结果。

### 2. Java 后端接收请求 – 任务初始化
- **接口层**：Spring Boot Controller 接收 JSON，进行 JWT 解析与权限校验（RBAC 鉴权，确认用户有权使用该功能）。
- **生成任务**：
  - 创建全局唯一 `taskId`（例如 UUID）。
  - 初始化任务记录到 MySQL 表 `task`：
    - `task_id`, `user_id`, `query_text`, `status = 'INIT'`, `create_time`, `update_time`。
  - 将原始请求和 `taskId` 封装为任务对象（JSON）。
- **状态变更**：`INIT` → 表示 Java 已收到请求，尚未发送到 RabbitMQ。

### 3. 发送消息到 RabbitMQ – 进入排队
- **动作**：Java 将任务 JSON 发送到 RabbitMQ 的交换机（例如 `travel.exchange`，路由键 `task.queue`）。
- **确认机制**：使用 **Publisher Confirms** 确保消息成功到达 Broker。
- **状态更新**：发送成功后，更新数据库 `status = 'QUEUED'`（排队中）。
- **可选**：在 Redis 中缓存任务状态和超时控制（如设置 60 秒 TTL，超时则触发降级）。

### 4. Python 后端消费与处理 – 状态流转
- **Python 消费者**：从 RabbitMQ 队列获取消息，手动 ACK（处理完后再确认）。
- **开始处理时**：
  - Python 通过 RabbitMQ 回传一个**进度消息**给 Java（例如 `{ "taskId": "xxx", "step": "start", "message": "大模型开始分析" }`）。
  - Java 监听 RabbitMQ 的**结果队列**（同时用于接收进度和最终结果），收到进度后：
    - 更新数据库 `status = 'PROCESSING'`（如果之前是 QUEUED）。
    - 通过 WebSocket 推送给前端（例如推送“正在调用 AI 规划行程…”）。
- **具体 Python 内部步骤**（对 Java 而言是黑盒，但会分阶段回传进度）：
  - **意图识别**：调用大模型分析用户查询，识别目的地、天数、偏好。
  - **工具调用**：如需天气、景点、美食数据，执行 RAG 检索或调用外部 API。
  - **行程生成**：模型生成结构化旅游计划（包含每日行程、交通建议、美食推荐等）。
  - **格式化输出**：将结果整理成 Java 约定好的 JSON 结构。

### 5. 结果回传与完成
- **Python 完成**：将最终结果 JSON 发送到 RabbitMQ 的结果队列（与进度消息同队列或不同队列均可，但需区分类型）。
- **Java 消费者（结果处理）**：
  - 接收消息，校验 `taskId`。
  - 使用 Jackson 反序列化为 Java 对象。
  - **持久化存储**：
    - 更新主表 `task`：`status = 'COMPLETED'`, `complete_time = now()`。
    - 将结果拆解存入细节表：行程表 `trip_detail`、每日计划表 `daily_schedule`、景点表 `attractions`、美食表 `foods` 等（结构化存储便于后续查询和修改）。
  - **WebSocket 推送**：根据 `taskId` 找到用户的 WebSocket 会话，将最终结果 JSON 推送给前端。
- **前端展示**：小程序接收后渲染为旅游规划文字、景点卡片、地图路线、天气信息等。

### 6. 异常处理与状态回滚
若任意环节出错，进入 `FAILED` 状态并记录错误原因：

| 异常场景                      | 触发条件                                          | Java 动作                               | 最终状态 |
| ----------------------------- | ------------------------------------------------- | --------------------------------------- | -------- |
| 消息发送失败                  | RabbitMQ 不可用或 Publisher Confirm 超时          | 重试 3 次，仍失败则更新数据库           | `FAILED` |
| Python 执行超时               | Redis 中的超时 TTL 过期（如 60 秒未收到任何回传） | 主动更新状态，关闭 WebSocket 并通知前端 | `FAILED` |
| Python 返回格式错误           | JSON 反序列化失败                                 | 记录错误日志，不更新细节表              | `FAILED` |
| AI 调用失败（如模型返回错误） | Python 侧捕获异常，发送 `{"error": true}` 消息    | 收到错误标记，更新状态                  | `FAILED` |
| 死信队列处理                  | 消息重试超过 3 次仍失败                           | 移入死信队列，人工介入                  | `FAILED` |

### 7. 全链路日志追踪
- 在 Java 接收请求时，生成 `traceId` 并放入 SLF4J MDC。
- 通过 RabbitMQ 消息头传递 `traceId` 给 Python。
- Python 在日志中同样打印该 `traceId`。
- 任何错误或状态变更都会附带 `traceId` 记录到数据库日志表，方便问题排查。

## 状态机总览
| 状态       | 触发动作                      | 存储位置                             |
| ---------- | ----------------------------- | ------------------------------------ |
| INIT       | Java 收到请求，写入数据库     | MySQL task 表                        |
| QUEUED     | 消息成功发送到 RabbitMQ       | MySQL task 表 + RabbitMQ 队列中      |
| PROCESSING | Python 开始处理并回传进度消息 | MySQL task 表（同时 WebSocket 推送） |
| COMPLETED  | Java 成功解析并持久化最终结果 | MySQL 多张表 + WebSocket 推送        |
| FAILED     | 任何环节出现不可恢复的错误    | MySQL task 表 + 错误日志表           |

## 关键组件交互时序图（文字描述）
1. 前端 → Java：HTTP POST (含 JWT)
2. Java → DB：INSERT task (INIT)
3. Java → RabbitMQ：发送任务消息
4. Java → DB：UPDATE task (QUEUED)
5. Java → 前端：HTTP 200 立即返回 taskId
6. 前端 → Java：WebSocket 连接，订阅 `/user/{userId}/task/{taskId}`
7. RabbitMQ → Python：消费者获取消息
8. Python → RabbitMQ：发送进度消息（step=start）
9. Java（监听器）← RabbitMQ：收到进度
10. Java → DB：UPDATE task (PROCESSING)
11. Java → 前端（WebSocket）：推送进度信息
12. Python 内部处理（可能多次回传中间进度，重复 8-11）
13. Python → RabbitMQ：发送最终结果 JSON
14. Java（监听器）← RabbitMQ：最终结果
15. Java → DB：UPDATE task (COMPLETED) + 写入细节表
16. Java → 前端（WebSocket）：推送完整旅游计划
17. 前端渲染展示

以上流程完整覆盖了用户请求从产生到结束的状态变化，并体现了 RabbitMQ 解耦、WebSocket 实时推送、MySQL 持久化及异常处理机制。
